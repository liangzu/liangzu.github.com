SymbolTable.h/.cpp

显然这就是符号表所在之处了。需要说明的是，符号表首先是一个栈(Stack),栈顶代表当前的作用域(Scope),退出、进入作用域就相当于对栈顶元素的操作了。 栈里的元素是 std::map, 意思是每个作用域都有一个自己的符号表。

这份数据结构的接口是:

void EnterScope()
进入作用域。
void ExitScope()
退出作用域。
Lookup(Symbol s)
从栈顶开始查找符号 s, 搜索整个作用域。
Probe(Symbol s)
查找符号 s, 只搜索当前作用域。

里边还定义了 EmptyScopeException 异常类, 用来发现对符号表的错误操作的情况。

最后需要说的是，其实在.l和.y文件中遇到某些token的时候可以直接把他们放进符号表。可是我暂时还没来得及搞这个。

lex.l

这是flex的文件，里面没什么好说的，现在我暂时没做错误处理，也没有把常量放到符号表，还没有处理STRING常量，这个挺麻烦的……后头补全。

这里面没有 SYS_PROC 、 SYS_FUNC 这种奇葩的不得了的东西是因为没有必要。他们会和identifier一起被识别，而且和函数调用语法一致，只要在 做语义分析之前把它们加到符号表里面去就可以了。

使用方法： 

flex -o [output-filename] lex.l

syntax.ypp

这是bison语法文件，我改了一些老师给的CFG，比如去掉奇葩的 goto, label, 然后把其它的部分改的好写了很多，不过基本上没有改变它原本的含义， 写完后有5个shift-reduce冲突（其实是3个), 由于bison默认选择shift,而shift恰好是我们所希望的语义，所以没什么问题。

修整后的语法放在 文末 ，大家应该把它和老师给的那个对比一下。由于我在bison文件里手动指定优先级，所以expr的cfg不会产生歧义。

使用方法： 

bison syntax.ypp -v -g -d

注:

  • .ypp的意思是生成cpp文件，而不是c文件(.y)。
  • -v的意思是verbose, 这样它会产生一个后缀为.output的文件，里面有详细的状态机的信息
  • -d是指同时生成头文件。此时 必须 在.hpp文件里手动包含 Tree.h ，因为在.ypp文件里写的 #include "Tree.h"会被放到.cpp里。
  • -g是指生成.dot文件，用来画图的，dot是一门简单的脚本语言，对此感兴趣的可以搜 graphviz （软件名）。

Tree.h/.cpp

这就是语法树了，写语法树是比较简单但是又特别繁琐的事情。很多都是重复的工作。

我在这里简单说明一下它的结构。由于语法树是根据文法写的，所以这里我也以文法为例介绍我的做法。 

for_stmt: FOR IDENTIFIER ASSIGNMENT expr TO expr DO stmt

以for循环的语法为例，这里FOR, ASSIGNMENT是为了表示程序结构的，语法树完全不需要存储他们的信息。

而我对应的代码是: 

enum class Direction{TO, DOWNTO};
struct For: Statement
{
    Ptr<Symbol> id;
    Ptr<Expr> start;
    Direction direction;
    Ptr<Expr> end;
    Ptr<Statement> stmt;

    For(const char* i, Ptr<Expr> s, Direction d,
        Ptr<Expr> e, Ptr<Statement> st)
            :
            id{new Symbol(i)}, start{s}, direction{d},
            end{e}, stmt{st}
    {}
};

需要注意的是如下几点:

  • id, start, direction, end, stmt 这成员变量分别代表文法右边的需要存储的信息。
  • 成员变量的类型基本上都是指针（此处direction不是），因为在语法树中，有些结点是前置声明的，我们没法在它定义前知道它的大小。

另外，值类型的变量在继承层次中向上转化就被切片了，而且后面做语义分析和代码生成时候值类型根本没法利用虚函数。

再举个例子。 cfg: 

stmt: assign_stmt
      | proc_stmt
      | compound_stmt
      | if_stmt
      | repeat_stmt
      | while_stmt
      | for_stmt
      | case_stmt

对于这样的语法，可以在 stmt 结点里面搞一个枚举，记录所保存结点的类型，然后访问它的时候一遍又一遍的问它“你是 for_stmt 吗"，再采取相应的动作。 但是这样 stmt 就会变得很“臃肿”，从美学角度上来说就很难看了。所以一般的（我所采用的)做法是： 

struct Assign: Statement{};
struct Proc: Statement{};
struct Compound: Statement{};

这样就想要操作 stmt 就要用虚函数，在各个子类里实现同样的方法(比如 CodeGeneration, SemantChecker)，由于每个子类的成员都是清晰的，所以我们要做什么也很清楚。
