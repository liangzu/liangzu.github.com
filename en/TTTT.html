Local variables can be shared between threads

We all know that global and static variables can be shared between threads, but some tricky programs share their local variables. For instance: 

#include <pthread.h>
#include <stdio.h>

void* thread(void* args);

char **pointer;

int main () {
    int i;
    pthread_t tid;
    char *str[] = {
        "Hello, Thread!",
        "Thread, Hello!"
    };

    pointer = str;

    pthread_create(&tid, NULL, thread, NULL);

    pthread_join(tid, NULL);
}

void* thread(void* args) {
    printf("Use Local Variables: %s", pointer[0]);

    return NULL;
}

The global variable pointer points to a local variable str in main thread, and thus other threads can access it.

Thread safety consideration

How do we know whether a function is thread-safe or not? CSAPP has given a clear definition:

The function that doesn't protect the shared variables
Thread-unsafe.
The function that returns a pointer to a static variable
Thread-unsafe.
The function whose returned value depends on the previous returned value
Thread-unsafe
The function which calls other thread-unsafe functions
Depends.

Scheduling shared resources by semaphore

CSAPP describes a concurrent queue to illustrate this idea.

The code is as following(can not be run): 

#include <semaphore.h>
#include <stdio.h>
#include <stdlib.h>

typedef struct {
    int *buf;
    int n;
    int front;
    int rear;
    sem_t mutex;
    sem_t slots;
    sem_t items;
} sbuf_t;


void sbuf_init(sbuf_t* sp, int n) {
    sp->buf = calloc(n, sizeof(int));
    sp->n = n;
    sp->front = sp->rear = 0;
    sem_init(&sp->mutex, 0, 1);
    sem_init(&sp->slots, 0, n);
    sem_init(&sp->items, 0, 0);
}

void sbuf_insert(sbuf_t* sp, int item) {
    P(&sp->slots);   // Function P undefined...
    P(&sp->mutex);

    sp->buf[(++sp->rear) % (sp->n)] = item;
    V(&sp->mutex);
    V(&sp->items);
}
int sbuf_remove(sbuf_t* sp) {
    int item;
    P(&sp->items);
    P(&sp->mutex);

    item = sp->buf[(++sp->front) % (sp->n)];
    V(&sp->mutex);
    V(&sp->slots);
    return item;
}

Implementing user-level thread

EOPL3 introduces the concepts of continuation. Continuation is an abstraction of control context. Once we can manage and control the continuation of the computation, we can implement threads simply in the user-level.

If you are interested in continuation and thread, having a look at chapter 5 in EOPL.
